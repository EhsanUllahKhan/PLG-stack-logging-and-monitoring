# Grafana + Loki — Slack Alert Setup Guide

This guide covers how to configure Grafana to send container error logs to a Slack channel using Loki as the log data source.

---

## Prerequisites

- Grafana is running and accessible (e.g. `https://your-grafana-domain`)
- Loki is configured as a data source in Grafana
- Container logs are being scraped into Loki with labels `instance` and `container`
- A Slack incoming webhook URL

---

## Step 1 — Create the Slack Message Template

A custom template controls exactly how the Slack message looks.

1. In Grafana, go to **Alerting → Contact points**
2. Click the **Notification templates** tab
3. Click **+ Add notification template**
4. Set the **Name** to `slack-template`
5. Paste the following template content:

```
{{ define "slack-error-log.title" -}}
:rotating_light: ERROR: {{ .GroupLabels.alertname }}
{{- end }}

{{ define "slack-error-log.message" -}}
{{ range .Alerts -}}
*Folder:* {{ .Labels.grafana_folder }}

*Host:* {{ .Labels.instance }}
*Container:* {{ .Labels.container }}
*Log:*
```{{ .Annotations.description }}```
*Time:* {{ .StartsAt.Format "2006-01-02T15:04:05.000Z" }}
{{ end -}}
{{- end }}
```

6. Click **Save**

---

## Step 2 — Create the Slack Contact Point

1. Go to **Alerting → Contact points**
2. Click **+ Add contact point**
3. Fill in:
   - **Name:** `ai-dev-error-log` (or any name you prefer)
   - **Integration:** Select `Slack`
   - **Webhook URL:** Paste your Slack incoming webhook URL
   - **Username:** `Grafana Alerts`
   - **Title:** `{{ template "slack-error-log.title" . }}`
   - **Text body:** `{{ template "slack-error-log.message" . }}`
   - **Mention channel:** Select `@channel` (to notify everyone)
   - **Disable resolved message:** ✅ Enable (so Slack is not spammed when alerts resolve)
4. Click **Test** to verify the webhook works
5. Click **Save contact point**

---

## Step 3 — Configure the Notification Policy

The notification policy routes alerts to the correct contact point.

1. Go to **Alerting → Notification policies**
2. Click **Edit** on the default policy (root)
3. Set **Default contact point** to your contact point (`ai-dev-error-log`)
4. Click **+ Add nested policy** and configure:
   - **Matching label:** `__grafana_autogenerated__` = `true`
   - **Contact point:** `ai-dev-error-log`
   - **Group by:** `grafana_folder`, `alertname`
   - **Group wait:** `30s`
   - **Group interval:** `5m`
   - **Repeat interval:** `4h`
5. Click **Save policy**

---

## Step 4 — Create a Dashboard for Each Container

Each container needs a dashboard with a logs panel so alert rules can be linked to it.

1. Go to **Dashboards → + New dashboard**
2. Click **+ Add visualization**
3. Select **Loki** as the data source
4. Switch to **Code** mode in the query editor
5. Enter the query (replace `instance` and `container` values):

```
{instance="example.com", container="discovery_worker"} | decolorize
```

6. In **Panel options**, set the **Title** to the container name (e.g. `cloudwhisper-discovery-worker-logs`)
7. Under **Options**, enable:
   - **Show time**: On
   - **Wrap lines**: Off
   - **Pretty print**: On
8. Set the default **time range** (top right) to `Last 30 seconds` or a narrow window
9. Click **Save dashboard** and name it matching the container

> Repeat for each container you want to monitor.

---

## Step 5 — Create an Alert Rule for Each Container

Alert rules watch for ERROR logs and fire when found.

1. Open the container's dashboard
2. Click the panel menu (three dots) → **Edit**
3. Click the **Alert** tab → **+ New alert rule**
4. Configure the alert:

   **Section A — Query and alert condition**
   - Switch to **Code** mode
   - Enter the Loki query:
     ```
     topk(1, count_over_time(({instance="example.com", container="discovery_worker"} |= "ERROR" | decolorize | regexp `(?P<log_message>.+)`)[1m]))
     ```
   - Set **Input** to the query result
   - Set **IS ABOVE** threshold to `0`

   **Section B — Alert evaluation behavior**
   - **Folder:** Set to your folder (e.g. `example.com`)
   - **Evaluation group:** Choose or create a group (e.g. `CloudWhisper`)
   - **Evaluate every:** `1m`
   - **For:** `0s` (fire immediately on first match)
   - **Configure no data and error handling:**
     - No data: `OK`
     - Error: `OK`

   **Section C — Annotations and labels**
   - **summary:** `ERROR in {{ $labels.container }}`
   - **description:** `{{ $labels.log_message }}`

   **Section D — Notifications**
   - **Contact point:** Select `ai-dev-error-log`

5. Click **Save rule and exit**

> Repeat for each container, updating `instance` and `container` values in the query.

---

## Step 6 — Verify the Alert

To confirm the full pipeline is working:

1. Go to **Alerting → Alert rules**
2. Find your rule — it should show **Normal** state (no current errors) or **Firing** if errors exist
3. To manually test: go to **Alerting → Contact points**, find `ai-dev-error-log`, click **Send test notification**
4. Check your Slack channel for the notification

**Expected Slack message format:**
```
@channel
🚨 ERROR: cloudwhisper-discovery-worker
Folder: example.com

Host: example.com
Container: discovery_worker
Log:
  11:57:22 | ERROR    | 📦 CELERY | Line :216 | Credential validation failed ...
Time: 2026-04-09T11:57:40.000Z
```

## Loki Query Reference

Replace `instance` and `container` values with your actual host and container name.

### All logs — Dashboard panel (shows INFO + WARNING + ERROR)

```
{instance="example.com", container="capability_executor_worker"} | decolorize
```

### Error logs only — Alert rule query

```
topk(1, count_over_time(({instance="example.com", container="capability_executor_worker"} |= "ERROR" | decolorize | regexp `(?P<log_message>.+)`)[1m]))
```

**How this query works (as shown in Grafana query builder):**

| Step | Expression | What it does |
|---|---|---|
| 1 | `{instance="example.com", container="capability_executor_worker"}` | Fetch all log lines matching these labels |
| 2 | `\|= "ERROR"` | Keep only lines that contain the string `ERROR` |
| 3 | `\| decolorize` | Strip ANSI color codes for clean readable output |
| 4 | `\| regexp \`(?P<log_message>.+)\`` | Extract the full log line as a label called `log_message` |
| 5 | `count_over_time(...[1m])` | Count matching lines in the last 1 minute |
| 6 | `topk(1, ...)` | Keep only the single most frequent error series (avoids duplicates) |

### Why `|= "ERROR"` before `decolorize`?
Container logs contain ANSI color codes (e.g. `\x1b[31mERROR\x1b[0m`). Filtering with `|= "ERROR"` on the raw line works because the word `ERROR` is always present as a literal string. Decolorize is then applied to strip the color codes for a clean readable `log_message`.

## Annotation Template Variables

| Variable | Value |
|---|---|
| `{{ $labels.container }}` | Container name (e.g. `discovery_worker`) |
| `{{ $labels.instance }}` | Host/instance (e.g. `example.com`) |
| `{{ $labels.grafana_folder }}` | Alert folder name |
| `{{ $labels.log_message }}` | The actual log line extracted by regexp |

## Troubleshooting

| Problem | Cause | Fix |
|---|---|---|
| Log shows `{{ .log_message }}` literally | Wrong template syntax | Change to `{{ $labels.log_message }}` in annotation |
| Log shows `"level": "ERROR"` (JSON fragment) | Container logs in JSON format | Adjust regexp to extract the `message` field from JSON |
| Alert shows `No log content available` | `noDataState` set to `NoData` | Set **No data** handling to `OK` in alert rule |
| Alert never fires | `\| ERROR` regex filter not matching ANSI logs | Use `\|= "ERROR"` (before decolorize), not regex |
| No Slack message despite alert firing | Routing policy not wired to Slack contact point | Check notification policy routes to `ai-dev-error-log` |
| Duplicate log entries per alert | No `topk` in query | Wrap query with `topk(1, count_over_time(...))` |
